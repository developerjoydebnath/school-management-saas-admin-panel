# Internationalization (i18n)

## Cookie-Based next-intl Routing

Do not use path-based routing (e.g., `/en/dashboard`) or `[locale]` dynamic segments in the `src/app` directory. Localization is driven entirely by the `NEXT_LOCALE` cookie, handled automatically in `src/i18n/request.ts`.

```tsx
// ✅ Correct
// App directory remains clean: src/app/(protected)/students/page.tsx
// URLs remain clean: /students

// ❌ Wrong
// src/app/[locale]/(protected)/students/page.tsx
// URLs: /en/students
```

**Why:** It keeps URLs clean and shareable. It drastically simplifies the `src/app` directory by removing the need for a root `[locale]` dynamic segment, and avoids complex Next.js routing middleware logic for language redirects.

## Feature Namespacing

Translation JSON files (`en.json`, `bn.json`) must group strings by the domain feature or module namespace, rather than being flat or grouped by UI element type. Components must initialize translations with their specific namespace. If a component needs shared strings, initialize a second hook for the `Common` namespace.

```tsx
// ✅ Correct (in en.json)
{
  "Classes": {
    "title": "Classes",
    "addClass": "Add Class"
  },
  "Common": {
    "delete": "Delete",
    "save": "Save"
  }
}

// ✅ Correct (in component)
const t = useTranslations("Classes");
const tc = useTranslations("Common"); // For shared strings

return (
  <div>
    <h1>{t("title")}</h1>
    <button>{tc("save")}</button>
  </div>
);

// ❌ Wrong (in component)
const t = useTranslations();
return <h1>{t("Classes.title")}</h1>;
```

**Why:** It prevents the global translation files from becoming massive, unmanageable flat lists of keys. By scoping translations to the domain feature, keys don't collide, and developers know exactly where to find or add strings for the module they are working on.

## Dynamic Data Fallback

When rendering dynamic translated strings that come from the database (e.g., an object like `{ en: "Class 1", bn: "ক্লাস ১" }`), you must use `getLocalizedName(data, locale)` from `@/shared/utils/localization`.

```tsx
// ✅ Correct
import { getLocalizedName } from "@/shared/utils/localization";
const locale = useLocale();

return <h1>{getLocalizedName(cls.name, locale)}</h1>;

// ❌ Wrong
const locale = useLocale();
return <h1>{cls.name[locale]}</h1>;
```

**Why:** Data stored in the database might not always have translations for every language. If you try to render `cls.name[locale]`, it will show empty (or crash) if the translation is missing. `getLocalizedName` safely falls back to English, preventing broken UI.

## Form Localization

When building creation and update forms, strictly limit localization to the form's **Title** and **Action Buttons**.

```tsx
// ✅ Correct
<DialogTitle>{t("createPermissionTitle")}</DialogTitle>
<InputField label="Permission Name" placeholder="e.g. View Classes" />
<Button>{t("createPermission")}</Button>

// ❌ Wrong
<DialogTitle>{t("createPermissionTitle")}</DialogTitle>
<InputField label={t("permissionNameLabel")} placeholder={t("permissionNamePlaceholder")} />
<Button>{t("createPermission")}</Button>
```

**Why:** Localizing every single form input label and placeholder creates massive overhead in translation files, slows down development, and reduces maintainability. Keeping data-entry fields in the primary business language (English) while localizing major actions maintains a balance between usability and developer experience.
